# Lessons

- When a missing tool or interpreter blocks environment setup, verify common versioned binaries explicitly and be prepared for the user to have installed one outside the current shell path assumptions. For this repo's Python venv, prefer checking `python3.10` before concluding the supported interpreter is unavailable.
- Do not infer a domain concept from a test name or proving infrastructure alone. In this repo, a proof over `cairo_pie_10_transfers.zip` is not evidence of a "private transaction"; verify the transaction semantics in the test input and code path before labeling it.
- When the user asks for a repeatable local test, prefer recording deterministic fixtures for offline replay over relying on live `latest` chain state. In this repo, that means using the existing `rpc_records` harness and saving the full RPC interaction set under `crates/starknet_transaction_prover/resources/rpc_records`.
- When the user asks for a fully offline test, capture every dynamic input in-repo before stopping: the request payload, the block-selection behavior, and the recorded RPC transcript needed for replay.
- When a recorded e2e transaction fixture targets a non-default Starknet network, encode that chain ID in the test itself instead of relying on a shell env var. Otherwise offline replay can fail with misleading signature errors even when the recorded RPC transcript is correct.
- Do not assume recorded RPC fixtures can be replayed in one global request order. In this prover, independent reads can be issued in different orders across runs because some class fetches come from nondeterministic iteration. Replay by consuming the first remaining matching interaction, not by enforcing transcript order across unrelated requests.
- When the user asks for a measurement or verification rerun, rebuild from a clean state first instead of trusting existing artifacts. In this proving stack, that means cleaning before the release `cargo test --no-run` build and only then collecting memory or proof-equivalence results.
